# EVALUATION PROTOCOL
## "How we prove the system works"

---

## A. METRICS — by layer

### NER (token + span)
- **Span-F1 (micro)** — exact match (start, end, type)
- **Span-F1 (macro)** — to expose minority-class weakness
- **Partial-match F1** — span overlap ≥ 0.5, type match — diagnostic only
- **Per-type F1** — 8 entity types
- **Calibration** — Expected Calibration Error (ECE) on dev

### Relation Extraction
- **Strict F1** — both spans correct, relation type correct
- **Boundary-relaxed F1** — type correct, spans within ±1 token
- **Per-relation F1** — 6 relations
- **Intra- vs cross-sentence F1** — guards against R2

### Rule + Ontology layer
- **Rule precision / recall** on seeded errors
- **SCOPE_GAP_WARNING precision** — % of warnings that are real gaps (sampled audit)

### End-to-end BOQ
- **Row precision / recall / F1** — match canonical rows vs ground-truth by fuzzy match
- **Completeness** — % of ground-truth items extracted (≥1 row)
- **Schema validity** — 100% rows pass Pydantic
- **Latency** — p50, p95 over golden 30-RFQ test set
- **Reproducibility** — same input + same seed → bytewise identical output

## B. BASELINES (we must beat each)

| # | Baseline | What it is | Why |
|---|---|---|---|
| B0 | Random | uniform tag | sanity floor |
| B1 | Gazetteer + regex | hand-listed materials, units | classical baseline |
| B2 | spaCy `en_core_web_trf` (no fine-tune) | off-the-shelf trf NER | bar for "doing nothing" |
| B3 | spaCy custom NER (CNN) | fine-tuned on our data | classical-deep baseline |
| B4 | BERT-base + linear head | minimal transformer fine-tune | shows BiLSTM-CRF contribution |
| B5 (ours) | BERT-base + BiLSTM + CRF | our proposed | must beat B4 by ≥ 2 F1 |

## C. ABLATIONS (each reported in final report)

| # | Ablation | Question |
|---|---|---|
| A1 | Remove CRF layer | Does CRF actually help vs simple argmax? |
| A2 | Remove BiLSTM, keep BERT-CRF | Does BiLSTM help on top of BERT? |
| A3 | Swap BERT → RoBERTa → DeBERTa-v3-small | Does encoder choice matter? |
| A4 | Remove rule validator | What % of errors do rules fix? |
| A5 | Remove ontology mapping | Does ontology improve type accuracy? |
| A6 | Sentence-level RE only vs ±3 sentence | Cost of cross-sentence relations? |
| A7 | Token classification vs span classification (for short spans) | Span model marginal? |

## D. ERROR ANALYSIS PROTOCOL (Step 4 deliverable)

Per model checkpoint:
1. Sample 100 dev errors stratified by type
2. Categorize: BOUNDARY / TYPE / MISSED / SPURIOUS / TOKENIZER / OCR
3. Per category: pareto chart + 5 representative examples
4. Action items: each category points to a fix (rule patch, data add, hyperparam change)
5. Notebook committed: `notebooks/04_errors_<date>.ipynb`

## G. STATISTICAL SIGNIFICANCE

Claims of "better than baseline" must report:
- Bootstrap CI (95%) on F1 (n=1000 resamples)
- p-value via paired permutation test on document-level F1
- Effect size when relevant

No "F1 0.87 vs 0.86 therefore better" without CIs.
