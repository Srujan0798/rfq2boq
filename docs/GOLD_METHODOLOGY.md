# Gold Methodology — what counts as a gold row, who may change gold, how the lock works

> **Authoritative for Phase 9.** Written by P0_02 (2026-07-06) under owner decisions D2 + D4.
> This document is the reference for `scripts/check_gold_provenance.py` and the
> `GOLD_LOCK.sha256` manifest. If anything here disagrees with code, this doc wins
> until the code is brought into line.

---

## 1. What gold IS

Gold is the **independent ground truth** the pipeline is measured against. It is
never produced by the pipeline. Two flavors live in this repo:

| Flavor | Location | Format |
|--------|----------|--------|
| Sentence gold (NER) | `data/real_rfqs/gold/swa_*.json` | `tokens[]` + BIOES `ner_tags[]` / `labels[]` |
| Row gold (fidelity) | `data/real_rfqs/gold/rows/*.rowgold.json` | one `entries[]` array of line items |

A gold **row** represents a real BOQ line item transcribed by a human from the
source document. The 10 sacred rowgold files carry a `method` field recording
how the gold was produced:

- `independent-xlsx-transcription` / `direct-xlsx-hand-transcription` →
  **independent** (human read the source XLSX and typed the row). Trusted.
- `pdfplumber-*` → **pipeline-derived** (the PDF extractor produced the gold).
  Self-comparison risk — `check_gold_provenance.py` warns on these. Converting
  them to independent gold (human annotation or non-pdfplumber extraction) is an
  open item, **not something any agent does unilaterally**.

## 2. What counts as a gold row — the D4 ruling (owner, 2026-07-06)

> **A row that is a section title only — it has an item number but no real
> quantity and no real unit, and exists solely to introduce child items — is
> NOT a BOQ line item and is excluded from gold.**

Applied to two rowgold files in P0_02:

- `02_isro_vssc.rowgold.json`: the `"Structure & civil"` entry (item_no 3,
  quantity `"0"`, unit `"no."`) was a section header, not a line item. Removed.
- `08_sael.rowgold.json`: the `"THERMAL INSULATION"` entry (item_no 1,
  quantity `"0"`, unit `"no."`) was a section header, not a line item. Removed.

Both files carry a top-level `ruling_D4` note field recording the ruling. The
`item_no` of remaining entries was **not** renumbered — `item_no` reflects the
original spreadsheet row provenance and must stay stable; gaps are expected.

A row with a real quantity/unit (even `"0"` when 0 is a genuine tendered qty) is
**not** a section header and stays. The distinguishing test is *title-only*: the
material text is a heading introducing children, not a thing being supplied.

## 3. Who may change gold — Rule 3 (owner-only)

**No agent creates, edits, extends, corrects, or stamps gold/annotations as
verified.** Full stop. The only authorized gold mutations in Phase 9 are:

1. **P0_02** (this task) — restore poisoned files to `b2546b6` (owner decision
   D2) and apply the D4 section-header ruling (owner decision D4). Done once.
2. **Owner direct edits** — Srujan may edit gold himself at any time (he is the
   owner). After any owner edit, the lock must be re-pinned (§5).
3. **P2_04** annotation review — the owner reviews draft annotations and stamps
   them `human_verified: true` with `reviewer: "srujan"`. Drafts remain
   `human_verified: false` until then.

`human_verified: true` is valid **only** with `reviewer: "srujan"`. Any other
reviewer string, or a `human_verified: true` with a fabricated reviewer, is a
hard failure (incident #7). The 9 legacy rowgold files carry
`human_verified: true` with no reviewer field — that predates the convention and
is a warn-only legacy item for the owner to backfill, not a hard failure.

## 4. How the lock works

`data/real_rfqs/gold/GOLD_LOCK.sha256` is a sha256 manifest of **every** file
under `data/real_rfqs/gold/` (recursive), one `<hash>  <path>` line per file,
sorted by path. The lock file itself is excluded from its own manifest.

`scripts/check_gold_provenance.py` calls `verify_gold_lock(gold_root)` which
recomputes every file's sha256 and compares against the lock. Any of these is a
**hard failure** (exit 1) with the offending path printed:

- **MODIFIED** — file exists but its hash differs from the locked hash.
- **DELETED** — file is in the lock but missing on disk.
- **ADDED** — file is on disk but not in the lock.

`tests/unit/test_gold_lock.py` exercises all four cases (pristine / modified /
added / deleted) plus the missing-lock case, using `tmp_path` fixtures so real
gold is never touched.

The lock is regenerated by:

```bash
find data/real_rfqs/gold -type f ! -name GOLD_LOCK.sha256 -exec shasum -a 256 {} \; \
  | sort -k2 > data/real_rfqs/gold/GOLD_LOCK.sha256
```

## 5. How a legitimate owner edit re-pins the lock

When the owner (Srujan) edits gold — a genuine correction, a new verified
annotation, or a ruling like D4 — the lock must be re-pinned in the same commit
that changes the gold. The procedure:

1. Owner makes the edit.
2. Regenerate the lock (command above).
3. Commit gold + lock together with a message that names the owner decision
   (e.g. *"gold: apply D4 ruling (owner-authorized 2026-07-06)"*).
4. `python3 scripts/check_gold_provenance.py` must report
   `✓ Gold lock INTACT (N files verified)` immediately after the commit.

An agent that needs gold changed does **not** perform these steps — it writes
the argument + evidence into its task report and the orchestrator/owner rules.
Changing gold to match pipeline output is incident #12; changing the ruler to
fix the measurement is incident #8/#10/#13. Both are absolute task failures.

## 6. Scope of the lock

The lock covers `data/real_rfqs/gold/` only. It does **not** cover:

- `data/annotations/` — draft annotation space (different task, P2_02–P2_04).
  Drafts are not locked because they are expected to change during review.
- `data/real_rfqs/corpus_manifest.json`, `data/real_rfqs/split_test.json` —
  separate integrity surfaces (corpus + frozen split), protected by their own
  checks (`check_split_leakage.py`, etc.).

## 7. The bottom line

Gold is owner-only. The lock makes unauthorized tampering loud. If
`check_gold_provenance.py` ever reports `Gold lock MISMATCH`, stop and
investigate — do not "fix" it by regenerating the lock without understanding
what changed and why.
