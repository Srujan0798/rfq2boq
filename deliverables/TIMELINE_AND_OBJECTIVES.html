⬢ The Problem

Manual BOQ extraction is slow, brittle, and inconsistent

A single 20-page tender takes 2–4 hours by hand. Different estimators produce different BOQs from the same input. Scope omissions create real contractual risk.

Manual today

  • 2–4 hours per 20-page tender
  • Error-prone — scope omissions
  • Inconsistent across estimators
  • Requires specialized domain knowledge

With this system

  • < 30 seconds per typical document
  • Deterministic, reproducible output
  • Confidence scores per item
  • CPWD DSR-compliant Excel
03 / 11
⬢ Objectives

Five concrete goals

  • Automate RFQ-to-BOQ conversion for Indian construction tenders
  • Cover 8 entity types: MATERIAL · QUANTITY · UNIT · LOCATION · DIMENSION · STANDARD · ACTION · GRADE
  • Integrate CPWD DSR 2023 rate library (501 line items, real rates)
  • Output in the formats estimators already use — Excel + JSON
  • Be honest about what works and what doesn't — no inflated claims

ML Model

BERT-BiLSTM-CRF handles open-vocabulary entities like MATERIAL and LOCATION

Pattern Rules

Regex + spaCy EntityRuler handle structured tokens — STANDARD, GRADE, UNIT

Ontology Validation

OmniClass + IFC enforce construction-domain type constraints

Extends Zhang & El-Gohary (2015) — Precision 0.969 / Recall 0.944 on construction regulatory documents.

⬢ Be Straight

Limitations

Data

Only 4 real Indian tender PDFs

Gold set is mostly synthetic. MATERIAL extraction is data-starved — that drives the 0.033 score.

Models

ARCBERT not yet integrated

Network-blocked from HuggingFace. Using SciBERT fallback. Expected +5–8% F1 once unblocked.

Languages

Hindi scaffolded, not deployed

IndicBERT module exists; weights pending download.

Scope

No SaaS, no multi-tenancy

Internal tool, not a product. By design.

The architecture is sound. The data is the limiter.
